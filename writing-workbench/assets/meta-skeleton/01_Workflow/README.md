# 01_Workflow — 写作流程层

## 角色

定义本项目"怎么走"——写作顺序、模块依赖、单模块调用模板、跨模块约定。它是每次开写前的**操作手册**，回答三个问题：

- 读者最后会按什么结构读这份成稿？
- 现在该写哪一块？
- 写这一块要按什么步骤、调用什么资源？

## 何时加载 / 何时写入

- **写入时机**：结构方案和 Modules 切分确定后立即写入。一旦本项目成稿结构、实际写作顺序、模块清单和依赖关系定下来，本层的"结构方案"、"写作顺序图"和"单模块调用模板"就成形了。
- **加载时机**：开新模块前查写作顺序与依赖；执行单模块写作时按调用模板逐步走。
- **更新时机**：发现实际依赖与初版顺序图不符时回写；否则不动。

## 本层至少包含

- **结构方案**：记录 Step 5 中比较过的 2-3 个方案，说明作者初步想法、目标读者偏好、领域惯例之间的取舍，并写明最终采用方案。默认文件名：`结构方案.md`。
- **写作顺序与依赖图**：列出全部模块、实际写作顺序、模块间上游依赖、就绪程度（数据/素材就绪程度）。**建议表格形式**（列：模块 / 上游依赖 / 就绪程度），便于扫读和后续更新。实际写作顺序不必等于成稿目录顺序。
- **单模块写作调用模板**：开工某模块时的标准 6 步流程（加载 Context → 加载本模块任务书 → 加载必读素材 → 写作 → 保存到 04_Drafts → 对照 05_Checklists 验收）。模板内**固化三种占位符约定**——`[DATA GAP]` / `[SOURCE]` / `[DEFER]`——由 Scaffold 阶段写入，作为本项目工程惯例的默认值；写作期遇缺立即标注，项目若需调整命名或扩充类别可在 00_Context 显式声明覆盖。

按需追加：

- 图表/脚注/引注的放置规则
- 跨模块的衔接交接约定（前后模块如何对接）
- 编译/导出阶段调用的外部 Skill 清单（如 typora-pandoc-zotero-thesis、docx、humanizer-zh）

## 与其他层的关系

本层只负责**流程**，不负责**内容**。流程要调用的内容在哪：

- 跨模块共享的规范 → 00_Context
- 当前模块要写什么、不许写什么 → 02_Modules 任务书
- 大体积素材 → 03_DataRoom
- 验收依据 → 05_Checklists

## 完整性约束

`结构方案.md` 必须写出最终选择。写作顺序图必须**覆盖 02_Modules 中的全部模块**。漏一个模块就意味着该模块没有依赖位次，AI 无法判断什么时候该开写它。
