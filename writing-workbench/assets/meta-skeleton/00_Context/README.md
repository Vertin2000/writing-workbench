# 00_Context — 预加载上下文层

## 角色

每次写作开工前**必读**的上下文。它是写作过程中跨模块共享、相对稳定、规模可控的"前置知识包"。

写每一个模块前，AI 都会先读这层下的全部文件。所以这层的内容应当：

- **稳定**：在整个写作周期内基本不变（变了就要回到本层修订，而不是放进单个模块）
- **小**：合计加载体量可控，不挤占模块本身的上下文预算
- **跨模块共享**：每个模块都需要它，否则不该放这里

## 何时加载 / 何时写入

- **写入时机**：项目启动后、动笔前。在工作流的 Inventory → Partition → Context 切分阶段确定本层有几份文件、每份叫什么。
- **加载时机**：每一次写模块、每一次验收，开工前都先加载本层全部文件。
- **更新时机**：当一项内容被发现是"跨模块共享 + 稳定"的，或某条规则需要全篇统一时，回写到本层；否则不动。

## 与其他层的关系

应该放这层：

- 项目立项目标 / 主线 / 论点 / 核心命题
- 目标读者画像与认知翻译标准
- 全篇通用的术语规范、标注体系、引用规范、措辞规范
- 全篇通用的方法论 / 指导思想 / pre-analysis plan / 世界观设定 / 风格指南

不该放这层：

- 仅一两个模块用得到的素材（→ 02_Modules 模块任务书的"必读材料"或 03_DataRoom）
- 大体积的原始素材（→ 03_DataRoom）
- 草稿与产出物（→ 04_Drafts）
- 验收清单（→ 05_Checklists）

## 完整性约束

本层与 02_Modules、03_DataRoom 之间的边界由 Partition 步骤确定。**Partition 给出的方案中，凡被分到 00_Context 的项，必须在本层有对应文件承载**。漏一项 = 该项在每次预加载里缺席 = 多个模块写作时认知不一致。

如果不确定一项该不该进本层，问自己：这条规则被违反时，是会让某一个模块出问题（→ 那项归 Modules），还是会让多个模块互相打架（→ 归本层）。
