# 05_Checklists — 验收清单层

## 角色

两层验收机制的承载处：

1. **模块级验收**：每个模块写完后逐项检查——交付物齐了吗？越界了吗？标注体系一致吗？
2. **全篇一致性验收**：所有模块写完后的横向检查——术语统一吗？论点不矛盾吗？衔接通顺吗？

## 何时加载 / 何时写入

- **写入时机**：Scaffold 阶段为每个模块生成一份验收清单（基于该模块任务书的五要素）。全篇一致性清单在所有模块任务书确定后写入。
- **加载时机**：
  - 单模块写完后加载该模块的验收清单
  - 全篇合稿前加载全篇一致性清单
- **更新时机**：写作过程中发现新的验收维度时追加。

## 本层至少包含

- 每个模块一份验收清单（02_Modules 任务书的「验收清单」字段以路径引用本层文件，本层是权威源）
- 一份全篇一致性清单（跨模块维度）

全篇一致性清单常见维度（按项目自行裁剪）：

- 术语一致性（同一概念全篇同一名称）
- 论点不矛盾（模块 A 的结论不被模块 B 推翻）
- 衔接与过渡（前后模块的首尾段落逻辑连贯）
- 引用与标注体系统一
- 格式规范统一（标题层级、编号、图表编号）
- 关键数字与事实复述（同一数字或事实可以在摘要、正文、复盘中重复，但每次出现应承担不同叙述功能；避免在不同章节用相同句式机械重复同一数字，必要时改写、合并、交叉引用或突出新的比较角度）

## 工作台健康检查

阶段性复盘时，可以用本层做一次轻量检查。重点看：

- `00_Context` 是否缺关键口径，或已经重到每次加载都拖慢写作。
- `00_Context` 中的 Context 文件是否都有可追溯的 `## 来源索引`。
- `01_Workflow` 是否保留结构方案、实际写作顺序和依赖关系。
- `02_Modules` 是否能直接指导写作，而不是只有空标题。
- `03_DataRoom` 是否有索引；素材有没有被模块任务书引用。
- `04_Drafts` 是否能追溯到数据、模块任务书和文风要求。
- `05_Checklists` 是否够用。够用即可，不追求复杂审计。

## 与其他层的关系

- 验收清单的**依据**来自 00_Context（全篇规范）和 02_Modules（模块任务书）
- 验收的**对象**是 04_Drafts 中的草稿
- 验收发现问题时，修改草稿或回溯修改规范/任务书——不在本层记录修改内容

## 完整性约束

每个 02_Modules 中的模块**都必须有对应的验收清单**。没有验收清单的模块 = 写完没人检查 = 质量不可控。
